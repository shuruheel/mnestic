# mnestic — Roadmap

mnestic is a maintained, independently-developed fork of [CozoDB](https://github.com/cozodb/cozo) — a transactional relational–graph–vector database that uses Datalog — focused on being a first-class **substrate for agentic memory**. See [`FORK.md`](./FORK.md) for provenance and attribution (all credit for the original design belongs to Ziyang Hu and the Cozo Project Authors), [`CHANGELOG-FORK.md`](./CHANGELOG-FORK.md) for everything shipped since the fork point, and the design specs under [`docs/specs/`](docs/specs/) for the deeper engineering plans behind the items here.

This document is the **public, forward-looking roadmap**: where the project is going and how to help.

## Our commitment

The single most important promise of this fork is that it is **actively maintained.** Upstream Cozo has been dormant since late 2024; mnestic exists to carry the engine forward for everyone building on it. Concretely that means a steady release cadence, responsiveness to issues, and a clear public direction — this doc.

## North star

Make the engine the best **substrate for agentic memory**: in *one embedded engine*, deliver

- **Hybrid retrieval** — vector (HNSW) + keyword (BM25) + graph traversal, fused in a single call;
- **Temporally-correct knowledge** — query memory as-of any point in time, and bi-temporally ("what did we believe *when*" — shipped in 0.10.0);
- **Incremental index maintenance** — upserts that keep vector/FTS indexes current without full rebuilds or long write locks;
- **Fast point-lookups co-located with semantic search**; and
- **The operational tooling** to run long-lived graph memory in production.

Every item is judged against that goal — and described as a general database mechanism, not in terms of any specific application.

## What's already shipped (through 0.12.0)

The agentic-memory *retrieval* foundation is largely in place. Highlights (see [`CHANGELOG-FORK.md`](./CHANGELOG-FORK.md) for detail):

- **Budgeted weighted traversal** (0.12.0): `BudgetedTraversal` — cheapest-first multi-seed expansion over non-negative weights under a required **global distinct-node budget** (`max_nodes`), an optional cost ceiling (`max_cost`), an **exact** hop bound (`max_depth`, layered per-`(node, hops)` labels — never depth-pruned Dijkstra), and an optional in-expansion admission gate (a gate relation + `admit:` predicate; a gated-out node spends no budget and never bridges). It emits `(node, cost, parent, depth)` — the settled shortest-path-tree fragment; parent pointers reconstruct any path in plain Datalog. Deterministic by construction, form-independent (positional edges ≡ a `graph:` cached projection, byte-for-byte), and interruptible (`:timeout` / `::kill`). Validated at the merge gate against its flagship consumer's production host-side BFS via a defined-equivalence oracle; measured there, one call over a cached projection replaced ~2·depth engine round-trips and ran **2–4× faster**. The `HybridSearch` expansion mode that seeds the traversal from the vector/FTS legs' own candidates (spec §9) is the designated follow-up — see the differentiators below. Spec: [`docs/specs/budgeted-traversal.md`](docs/specs/budgeted-traversal.md).
- **Cached graph projections** (0.11.0): `::graph create G { edges: knows, nodes: person }` names an in-memory adjacency that twelve graph algorithms reuse across queries via a `graph:` option, instead of re-scanning the edge relation and rebuilding a CSR on every call — measured 15–16× on a 400,000-edge graph for `PageRank` and `ConnectedComponents`. A projection is **always fresh** (a write to a source relation frees what was built from it; under continuous write churn the cache degrades to build-per-query rather than going stale), in-memory, with a 512 MiB LRU ceiling (`set_graph_projection_capacity`). Two long-standing gaps in the same corner closed with it: `PageRank` now takes an optional node relation, so vertices with no edges are ranked instead of silently dropped, and it warns instead of returning unconverged ranks when the `iterations` cap stops it short — **its default `iterations` is now 20, up from a non-convergent 10, a breaking result change; pass `iterations: 10` to restore.** Spec: [`docs/specs/graph-projection.md`](docs/specs/graph-projection.md).
- **Query interruptibility, budgets & a deterministic join reorder** (0.10.5): `::kill` and `:timeout` now actually interrupt a running query, and a query can carry a per-query wall-clock budget (`:timeout`, a per-call option, or a Db-wide default; effective deadline = the minimum of whichever are set). A **default-on, stat-free greedy join reorder** removes the N³ blow-up a naively-ordered conjunction — exactly what an LLM agent authors — falls into (measured 54.5× on the repro; results unchanged; opt out per-query with `:reorder written`; 0.10.7 corrects a tie-break so a partial composite-key prefix is no longer mistaken for a point lookup and pulled ahead of a selective atom on high-fan-out cyclic joins). An **opt-in** factorized `count()` rewrite counts an eligible single-clause `count()`-over-join without materializing the join. Specs: [`docs/specs/join-reorder.md`](docs/specs/join-reorder.md), [`docs/specs/cardinality-algebra.md`](docs/specs/cardinality-algebra.md).
- **Antichain / skyline aggregate + interval primitives** (0.10.1; built-in Pareto added 0.11.1): `register_bounded_meet_aggr` keeps, per group, the non-dominated (Pareto-frontier) set under a host-registered strict partial order (Rust-embedded-only), and `interval_overlaps` / `interval_coalesce` operate over half-open `[start, end)` list intervals. **0.11.1** adds the built-in `pareto_min` / `pareto_max` aggregates — native componentwise dominance over a numeric vector — reachable from *every* binding through plain `run_script` with no registration, covering the canonical skyline case that the host-closure form could not reach. Specs: [`docs/specs/antichain-bounded-meet.md`](docs/specs/antichain-bounded-meet.md), [`docs/specs/cozoscript-extensions.md`](docs/specs/cozoscript-extensions.md).
- **Bi-temporality — system-versioned (`TxTime`) relations** (0.10.0, the marquee feature): an engine-assigned transaction-time axis alongside Cozo's valid-time — commit-clock stamping, current-state reads by default, time-travel `@ (vt: ..., tt: ...)` and `:as_of`, existence-checking writes, `::history` / `::history_gc` (persisted floor) / `::evict` (audited hard deletion), and `:reconcile` declarative belief revision. "What did we believe at time *T* about period *Y*" — in-engine, in an embedded database. Spec: [`docs/specs/bitemporality.md`](docs/specs/bitemporality.md).
- **Custom aggregates + top-k proofs** (0.10.0): `register_custom_aggr` for domain-specific absorptive combines in recursive rules, and `min_cost_k([payload, cost], k)` — a bounded-meet aggregate returning the k best derivations per answer with their evidence (Scallop-style approximate top-k). Spec: [`docs/specs/provenance-semirings.md`](docs/specs/provenance-semirings.md).
- **Read-only Cypher query surface** (alpha; behind the off-by-default `cypher` Cargo feature) — translates a subset of openCypher (`MATCH` / `WHERE` / `RETURN` with `DISTINCT` & aggregates / `ORDER BY` / `SKIP` / `LIMIT`; true bag semantics; null-aware `WHERE`; edge-isomorphism) to CozoScript, so the engine is easy to evaluate without learning Datalog first. Read interop only; Datalog stays the native, full-power language. API: `run_cypher` / `cypher_to_script`. The PyPI wheel ships without it for now (build `--features cypher`). Spec: [`docs/specs/cypher-read.md`](docs/specs/cypher-read.md).
- **One-call hybrid retrieval** (`HybridSearch`) with **Reciprocal Rank Fusion** and **MMR** diversity reranking.
- **Native 3-way fused recall** — vector + full-text + *k*-hop graph proximity fused in a single query (typed `GraphLeg`).
- **Okapi BM25 full-text scoring** with `k1`/`b` tuning, multi-term OR-summation, and O(1) average-document-length.
- **Per-leg fusion detail** — reconstruct exactly why each result was retrieved.
- **LangChain & LlamaIndex integrations** on PyPI (`mnestic`, `langchain-mnestic`, `llama-index-vector-stores-mnestic`).
- **Fast, non-blocking HNSW index builds** — a flat in-RAM parallel build (~15× faster on a 40k×384 corpus) that doesn't stall readers.
- **Snapshot read path** + batched neighbour fetch (`multi_get`) for read-only queries.
- **Corruption resilience & upgrade-safe catalogs** — `::repair_corrupt` and tolerant index builds, so one bad row never makes a database unopenable; and (0.10.6) forward-compatible relation catalogs — a database written by a pre-0.10.0 build keeps opening across engine upgrades (the 0.10.0 bitemporality field addition had made pre-0.10.0 catalogs undeserializable).
- **Planner & DX fixes** — equality-pushdown keyed lookups (~28× on point queries), keyword-prefixed identifier parsing, ULID functions.

## What's next

Tiered by value and how ready each item is. This is direction, not dated commitments.

### Near-term — correctness, cadence & developer experience

- **Steady release cadence** with clear migration notes.
- **Closing long-open upstream issues** under active maintenance — e.g. Sled backend `del()` correctness, SQLite-backend performance (prepared statements / `WITHOUT ROWID`), and modeling ergonomics for tree-shaped / JSON-LD data.
- **Better onboarding** — worked examples and clearer modeling guidance, lowering the "modeling my data in Datalog" learning curve. (Binder-error *quality* graduates from here into the **agent-actionable errors** differentiator below — for an increasingly-agent authorship it is a safety-net feature, not onboarding polish.)
- **A reproducible temporal-belief workload in the public benchmark suite.** Ingest sources, derive top-k proofs (`min_cost_k`), introduce a contradiction and surface the non-dominated set (`pareto_min`/`pareto_max` or a registered dominance order), retract a source, `:reconcile`, and assert both the valid-time and the transaction-time answers. The bitemporal and provenance features shipped across 0.10.x; this is the artifact that proves they *compose* end-to-end — a deterministic, replayable answer to "what did we believe, and why, and what happens when a source is retracted." Honesty note that the workload also enforces: proof identity in a `min_cost_k` payload is a user convention, and `:reconcile` is whole-relation recompute — *programmable* proof-carrying recursion with recompute-based belief revision, not automatic lineage. If whole-relation recompute shows up as the measured bottleneck here, that is exactly the evidence a *scoped* `:reconcile` has been waiting for. A companion axis proves the *embedded* North Star rather than any one feature: run the same oracle-verified workload against a **server-based** knowledge engine to isolate the per-query network round-trip floor an in-process engine never pays, using the same correctness-first, result-set-not-row-count discipline the graph-analytics suite already applies — over the reasoning shapes this engine is actually for: a contested-binding skyline, **multi-answer queries** that must return several attributed readings of one question instead of silently merging them, and **scoped completeness** (a known-complete dead end vs an open one). Described as general query shapes, not as any single consuming application; the finding worth publishing is that an embedded Datalog engine expresses and answers all three with no fixed network floor — the North Star made measurable.
- **A planner regression suite in CI.** The 0.10.5 join reorder shipped default-on and introduced a pathological plan on one high-fan-out cyclic shape; it was found in the field and fixed in 0.10.7. A scheduled join-planner smoke (a small labelled-subgraph query set plus the reorder pass's own repro, asserting no query regresses past a wall-clock threshold) should catch that class before a release, not after. It should also pin the *recursive* workload — fixpoint reachability at increasing depth — which is the thing this engine is fastest at and therefore the thing a regression would hurt most. The first pieces are in-tree as of 0.12.0: a zero-data plan-shape gate that catches the 0.10.5-class reorder regression (`cozo-core/tests/planner_shape.rs`) and a nightly LSQB sf0.1 execution tier (`cozo-core/tests/lsqb.rs`, `.github/workflows/planner-guard.yml`); the recursive-workload pin is still to come. Until the suite is complete, the honest position is that we cannot responsibly enable another planner pass by default, and the opt-in factorized-`count()` rewrite stays opt-in for the same reason: not because it fails, but because we have nothing standing to catch it if it ever does.

### The differentiators we're building

- **`::explain` as an advisor, not just a plan dump.** mnestic deliberately has no cost-based optimizer — plans are deterministic and the query author is the optimizer. That makes it the engine's job to *tell you what it sees*: which atom is a full scan, where a residual Cartesian product survives, and when a `count()` would factorize (the last two already ship). Extending this into a coherent diagnostic surface is the single cheapest thing we can do for query authors — and it matters more, not less, when the author is an LLM agent that can read the advice and rewrite the query.

  One sharp edge worth documenting, since it has already misled us: **the factorization advisory declining does not mean your query is cyclic.** The rewrite bails out on the *first* disqualifying atom it meets — any predicate, any negation, any rule or search atom — and that check runs before the shape analysis does. So "declined" can mean "cyclic," but it can equally mean "you used a filter." Reporting *which* condition fired, and analysing the shape even when an earlier condition already disqualified the rewrite, is exactly the kind of thing this item is for.

  The advice also needs somewhere to *go*. Today the residual-Cartesian finding reaches a `log::warn!` and nothing else — no typed output, no counter, no hook — and `::explain`'s columns are an internal debug format with no stability guarantee, so you cannot assert on them in CI. A downstream project shipped a full-scan traversal for months while the engine was warning about it on every call. A stable, structured diagnostics surface (a typed `warnings` column, or a `Db`-level accessor) is part of this item, not a follow-up to it.
- **Agent-actionable errors — the safety net an untyped engine owes its authors.** Positional tuples carry no schema, so nothing catches a mis-shaped binding at author time the way a typed query engine's compiler does; the discipline lives in convention and in whatever the engine says back. That makes the *quality* of what it says back a feature, not onboarding polish. A binder error that names the exact unbound variable, the arity it expected, and the atom the mismatch came from is what an LLM agent reads and acts on directly — the role a typed language's compile-time complaint plays, delivered through errors instead of through types. This is the deliberate other half of the untyped trade: MindGraph layers structural types on top, and the engine owes its authors — increasingly agents — errors precise enough to self-correct without them. Two surfaces concretely: precise binder diagnostics (which variable, which atom, expected vs actual shape), and its plan-time sibling above — the stable, structured `::explain` advisory an agent can assert on rather than scrape from a debug dump.
- **A `HybridSearch` budgeted-expansion mode** — the retrieval integration of the budgeted weighted traversal that shipped in 0.12.0 (the original 0.12 scoping paired the two; the `BudgetedTraversal` FixedRule landed first, and this is its designated follow-up, targeted at 0.12.1). `HybridSearch` grows an expansion mode that **seeds the traversal from the vector and keyword legs' own candidates** — fill the remaining budget with the cheapest graph context around what semantic and keyword recall found, with each admitted node's cost and parent reported in the per-leg detail. Today's `GraphLeg` requires every seed up front and ranks by unweighted hop distance; the expansion mode subsumes that, and it puts budgeted expansion inside the one-call `hybrid_search` surface the Python / LangChain / LlamaIndex integrations wrap, instead of requiring a hand-written Datalog rule around the FixedRule. The seam is pinned in the spec ([`docs/specs/budgeted-traversal.md`](docs/specs/budgeted-traversal.md) §9): the seeds input becomes the vector/FTS legs' own top-k rule, ranked into the fusion by ascending cost; the open design choices (deriving seed `initial_cost` from leg ranks; whether `GraphLeg` is deprecated or kept as the no-budget fast path) belong to this item.
- **Binding-reachable custom dominance orders.** The contested-set / skyline machinery ships in two forms with a seam between them: `pareto_min` / `pareto_max` (0.11.1) apply *componentwise numeric* dominance and are reachable from **every** binding through plain `run_script`, while `register_bounded_meet_aggr` (0.10.1) applies **any** strict partial order but only through a Rust host closure. A consumer whose dominance is a custom order that is *not* componentwise-numeric — an agent-specificity tie-break, a non-numeric strength preorder — therefore drops back to naive negation-over-a-self-join from Python or any other binding, with the efficient bounded-meet store sitting unused right beside it. Closing the seam is letting a caller name a dominance order the aggregate can use without linking Rust. The near form is documentation: a recipe for *encoding* common orders into the numeric vector the `pareto_*` builtins already take (rank offsets, lexicographic packing). The far form is the in-language dominance selector that [`docs/specs/antichain-bounded-meet.md`](docs/specs/antichain-bounded-meet.md) (Q3) and [`docs/specs/cozoscript-extensions.md`](docs/specs/cozoscript-extensions.md) (§3.5) deliberately deferred, whose stated trigger — "the first program needing two engine-owned algebras at once" — an external consumer modeling contested bindings from a binding rather than from Rust is the first real pull toward. Evidence-gated: the built-in skyline covers the numeric case today, so this waits on a concrete order that is provably *not* numeric-encodable before it opens the registration surface to the query language.
- **Stream passthrough rules into fixed-rule inputs.** A rule whose body is a single stored-relation atom — `edges[a, b] := *knows[a, b]` — is materialized into a temporary store before the fixed rule ever sees it. Passing the stored relation directly instead, `ConnectedComponents(*knows[a, b], *person[x, e])`, skips that copy and runs **~2.1× faster** on a 439k-edge graph, for identical results. Nobody should have to know this. The planner can see that the rule is a passthrough, and should stream it.
- **Extending the Cypher-read surface** — variable-length paths, `OPTIONAL MATCH`, `WITH`, and undirected relationships (today these return explicit not-yet-supported errors). Spec: [`docs/specs/cypher-read.md`](docs/specs/cypher-read.md).
- **Stored / named queries** — reusable, parameterized retrieval rules; also the substrate for a future compiled-plan cache.
- **A first-class ULID type** and sortable auto-keys (the scalar functions already ship; the type does not yet).
- **An official schema-migration tool** — versioned schema, diff, and rollback.
- **`LOAD FROM` Parquet/Arrow** + zero-copy Arrow export, for clean handoff with Python/Rust data pipelines.

### Performance at scale (evidence-gated)

Pursued when a real workload demonstrates the need, always with before/after measurements (the project's baseline-first rule):

- Compiled-plan / prepared-statement caching for high-frequency point reads.
- **Restoring the `!=` inclusion-exclusion factorized-count rewrite.** The engine's automatic `count()`-over-join rewrite (0.10.5) declines on any body containing a `!=`. An extension handling that case via inclusion-exclusion — `count(body ∧ A≠B) = count(body) − count(body[A:=B])` — was written and then withdrawn before release, because it miscounted on mixed `Int`/`Float` data: `!=` compares numerically (so `1 != 1.0` is false), while the storage order never equates `Int(1)` with `Float(1.0)`, so the subtracted "diagonal" misses exactly those pairs. The fix is to admit the rewrite only when the two inequality operands are provably the same type (or non-numeric) — the same guard the engine already applies to its equality-pushdown optimization, for the same reason. This is the single largest measured win available on count-over-join workloads, and until it lands, [`docs/specs/cardinality-algebra.md`](docs/specs/cardinality-algebra.md) §3.3 documents the hand-authored form.

- **Robust join execution without a cost-based optimizer** — Bloom/semijoin predicate transfer, which pre-filters relations before the join so that plan quality stops depending on getting the join *order* right ([Debunking the Myth of Join Ordering](https://arxiv.org/pdf/2502.15181): with semijoin Bloom pre-filtering, even random join orders land within ≤1.6× of optimal). This is the principled alternative to accumulating join-order heuristics, and it preserves the no-statistics, deterministic-plan design. **Honest status: we have no measured case for it yet, and we briefly thought we did.** A field benchmark's slowest query looked like the perfect motivation — an acyclic path whose inequality filter is applied late — until we noticed it is a `count()`. Semijoin reduction removes tuples that join with *nothing*; it does not spare you from enumerating matches that genuinely exist, and counting a billion real matches is what that query actually spends its time on. The fix there is the factorized-count rewrite above, not this. The two techniques are often both called "Yannakakis" and they are not the same thing: one is a full reducer, the other is bottom-up aggregation that never materializes the join. This item keeps its argument from first principles — plans authored by an LLM agent arrive in arbitrary join orders — but it is waiting for a workload that is *not* an aggregate before it gets built.
- Selectivity-tiered filtered vector search (efficient metadata-filtered ANN).
- Full-text scale headroom (compact posting-list storage + top-*k* pruning).
- Tunable/weighted fusion and graph-leg improvements, gated on retrieval benchmarks.
- Vector quantization with float rescore, gated on corpus-scale evidence.

## Non-goals (deliberate scope)

mnestic is an **embedded, single-node** engine specialized for agentic memory. To stay excellent at that, the following are intentionally out of scope:

- **Distributed clustering / replication / consensus** inside the engine.
- **Worst-case-optimal joins, and competitive performance on cyclic subgraph matching.** mnestic evaluates conjunctions with binary joins. On a *cyclic* pattern (a triangle, say) no join order avoids materializing intermediates above the output's worst-case size bound, so benchmarks built around cyclic subgraph matching — LSQB and its relatives — will show mnestic well behind engines with worst-case-optimal join operators, and no planner or reorder tuning changes that. We consider this a fair trade: agentic-memory workloads are point lookups, small neighbourhoods, vector search, and recursive rules, which are overwhelmingly acyclic; and the published evidence on WCOJ is genuinely mixed (large wins on triangles, ~9.6× *worse* on average for pure Generic Join in the Free Join evaluation). If a real agentic-memory workload ever presents cyclic queries, the entry point would be a runtime hash-trie operator gated by syntactic cyclicity — not a wholesale planner change. Until then, hand-factorize cyclic queries; [`docs/specs/cardinality-algebra.md`](docs/specs/cardinality-algebra.md) documents the patterns.

  **This is a scope boundary, not a general excuse — and we have caught ourselves misusing it.** When we first analysed the LSQB queries that time out, we labelled all of them "the cyclic ones." They are not. Re-measured with the timeout lifted, only two of the slow queries are genuinely cyclic; another is an *acyclic* count that is slow for an ordinary, fixable reason (see the factorized-count rewrite above), and one we had simply never determined the shape of. "Cyclic, therefore out of scope" is a claim that has to be earned per query. If a query of yours is slow, do not assume this bullet explains it; the honest answer is usually a filter, an index, or an aggregate that should have been factorized — and we would like to see the query.

  We also checked the one workload on our own roadmap that might have been cyclic and hot: the contested-set / antichain reasoning the dominance aggregate is built for. It is a **linear recursion feeding a per-group dominance fold** — acyclic. Pairwise contradiction detection is a star join — acyclic. Only maximal-clique enumeration over mutually-contradicting facts would be cyclic, and that is a different operator that nothing here asks for. So the exception above is not a placeholder we expect to trigger; it is a genuine boundary.
- **Multi-model breadth** (becoming a general document/time-series/KV store with many query languages) — the opinionated graph+vector+FTS focus is the point.
- **Data federation / virtualization** over external warehouses or lakehouses (mapping a schema onto external sources without copying). Agentic memory is copy-and-transform; that's a different kind of system.
- **Cypher *write* semantics** (read interop already ships as an alpha feature; full/write Cypher is not on the path).
- **CRDT multi-device sync** and **browser/WASM persistence** — real demand, but off this project's focus.

These can be revisited if a concrete agentic-memory need ever forces them, but they are not on the path today.

## How to contribute

Contributions are very welcome — mnestic is meant to serve the whole Cozo/Kùzu-refugee community, not just its primary downstream consumer.

- **Good first issues** are labeled in the tracker; the "near-term" items above are the best on-ramps.
- **Performance work must be baseline-first** — include before/after numbers; the repo has criterion benches to build on.
- **Tests:** keep the inherited engine tests green (they encode upstream semantics), and use the **SQLite** backend for any planner/stored-relation test (the in-memory backend uses a different join operator). See [`CONTRIBUTING.md`](./CONTRIBUTING.md) for the test-backend conventions.
- **Licensing:** mnestic is MPL-2.0. Preserve the original `Copyright … The Cozo Project Authors` headers on any file you modify.

If you're considering a larger feature (for example extending the Cypher-read surface, or building on the now-shipped bi-temporality — GC policy, audit tooling, benchmarks), open an issue to discuss the design first.

## Releases & versioning

- **SemVer on the 0.x line** — patch for fixes, minor for non-drop-in behavior/feature changes.
- Releases are published to **crates.io** (`mnestic`, `mnestic-rocks`) and **PyPI** (`mnestic`), with divergences recorded in [`CHANGELOG-FORK.md`](./CHANGELOG-FORK.md).
- Work is banked and released on a regular rhythm with migration notes — steady stewardship over churn.

## Relationship to MindGraph

mnestic is developed alongside, and consumed by, [MindGraph](https://crates.io/crates/mindgraph), a typed cognitive knowledge-graph library built on top of it. MindGraph is the engine's most demanding user and drives much of its hardening — but mnestic is a general-purpose database in its own right, and its roadmap is set to serve the broader community of developers building local-first and embedded AI memory.
